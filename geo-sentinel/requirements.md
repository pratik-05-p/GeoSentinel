# Requirements Document

## Introduction

GeoSentinel is a fully serverless web application that enables geology students, researchers, civil engineers, disaster management teams, and environmental consultants to perform preliminary landslide risk assessments using geological and environmental parameters. Users input seven key indicators — slope angle, rainfall intensity, soil type, vegetation cover, fault zone proximity, land use type, and drainage condition — and the system returns a scored hazard assessment (0–100), a risk category (Low / Moderate / High / Critical), a factor-level explanation, and actionable mitigation recommendations. All assessments are persisted for later review, and users can download a formatted PDF report. The platform is deployed on AWS using a fully serverless architecture: S3 + CloudFront for the frontend, API Gateway + Lambda for the backend, and DynamoDB for persistence.

---

## Glossary

- **Assessment**: A single landslide risk evaluation submitted by a user, containing input parameters, a computed hazard score, a risk category, explanations, and recommendations.
- **Assessment_Engine**: The Lambda-backed computation component that calculates the hazard score and derives risk category, explanations, and recommendations from a set of input parameters.
- **Assessment_History**: The persisted collection of past assessments for a user session or user account, stored in DynamoDB.
- **Dashboard**: The single-page application view that displays the risk gauge, risk category badge, hazard analysis card, recommendations panel, and assessment history table.
- **DynamoDB**: The Amazon DynamoDB table used to store assessment records.
- **Fault_Zone_Proximity**: The user-supplied distance (in kilometres) from the nearest known geological fault zone.
- **Hazard_Score**: A numeric value in the range [0, 100] representing the computed landslide susceptibility; higher values indicate greater risk.
- **Input_Parameters**: The seven user-supplied values required to perform an assessment: Slope_Angle, Rainfall_Intensity, Soil_Type, Vegetation_Cover, Fault_Zone_Proximity, Land_Use_Type, and Drainage_Condition.
- **Lambda**: An AWS Lambda function that implements the Assessment_Engine and all backend API logic.
- **Land_Use_Type**: A categorical value describing the current land use at the assessed location (e.g., Forest, Agricultural, Urban, Bare_Ground).
- **Mitigation_Recommendation**: A specific, actionable suggestion generated by the Assessment_Engine to reduce landslide susceptibility.
- **PDF_Report**: A downloadable document containing the Input_Parameters, Hazard_Score, Risk_Category, Risk_Explanation, and Mitigation_Recommendations for a given Assessment.
- **Rainfall_Intensity**: The user-supplied average or peak rainfall intensity in millimetres per hour (mm/h).
- **Risk_Category**: A qualitative label derived from the Hazard_Score: Low (0–24), Moderate (25–49), High (50–74), or Critical (75–100).
- **Risk_Explanation**: A structured textual breakdown identifying which Input_Parameters increased risk, which reduced risk, and an overall interpretation of the Assessment.
- **Slope_Angle**: The user-supplied angle of the terrain slope in degrees (°).
- **Soil_Type**: A categorical value describing the dominant soil classification at the assessed location (e.g., Clay, Sandy, Loam, Rocky, Silt).
- **User**: Any person accessing GeoSentinel through a web browser — including geology students, researchers, remote sensing learners, civil engineers, disaster management authorities, and environmental consultants.
- **Vegetation_Cover**: A percentage value (0–100%) representing the fraction of the assessed area covered by vegetation.

## Requirements

### Requirement 1: Input Parameter Collection

**User Story:** As a user, I want to enter geological and environmental parameters for a site, so that the system can evaluate the landslide risk for that location.

#### Acceptance Criteria

1. THE Dashboard SHALL provide an input form with seven fields: Slope_Angle, Rainfall_Intensity, Soil_Type, Vegetation_Cover, Fault_Zone_Proximity, Land_Use_Type, and Drainage_Condition.
2. THE Dashboard SHALL accept Slope_Angle as a numeric value in the range [0, 90] degrees, inclusive.
3. THE Dashboard SHALL accept Rainfall_Intensity as a numeric value greater than or equal to 0 mm/h and less than or equal to 1000 mm/h.
4. THE Dashboard SHALL accept Soil_Type as one of the following categorical values: Clay, Sandy, Loam, Rocky, Silt; any other value SHALL be rejected.
5. THE Dashboard SHALL accept Vegetation_Cover as a numeric value in the range [0, 100] percent, inclusive.
6. THE Dashboard SHALL accept Fault_Zone_Proximity as a numeric value greater than or equal to 0 kilometres and less than or equal to 500 kilometres.
7. THE Dashboard SHALL accept Land_Use_Type as one of the following categorical values: Forest, Agricultural, Urban, Bare_Ground; any other value SHALL be rejected.
8. THE Dashboard SHALL accept Drainage_Condition as one of the following categorical values: Good, Moderate, Poor; any other value SHALL be rejected.
9. IF a user submits the form with a missing, non-numeric (for numeric fields), out-of-range, or invalid categorical value for any field, THEN THE Dashboard SHALL display a descriptive validation error message adjacent to each specific invalid field that identifies the field name and states the acceptable range or permitted values; the form SHALL NOT be transmitted until all fields are valid.
10. WHEN a user submits the form and all seven fields pass validation, THE Dashboard SHALL transmit the Input_Parameters to the Assessment_Engine via the API Gateway over HTTPS and SHALL display a loading indicator on the submission control, disabling re-submission, until a response is received or the request times out after 30 seconds.
11. IF the API transmission fails (network error, HTTP 4xx, HTTP 5xx, or 30-second timeout) after a validated form submission, THEN THE Dashboard SHALL dismiss the loading indicator, re-enable the submission control, and display a descriptive error message that does not expose internal server details; the Assessment SHALL NOT be recorded in Assessment_History.

### Requirement 2: Hazard Score Computation

**User Story:** As a user, I want the system to compute a landslide hazard score from my inputs, so that I can understand the numeric level of risk at a site.

#### Acceptance Criteria

1. WHEN the Assessment_Engine receives a complete set of valid Input_Parameters, THE Assessment_Engine SHALL compute a Hazard_Score in the range [0, 100].
2. THE Assessment_Engine SHALL assign weighted contributions to each input parameter such that the combined weights of all parameters sum to 1.0.
3. THE Assessment_Engine SHALL produce a higher Hazard_Score for combinations of inputs that represent greater geological instability (e.g., steep slope, high rainfall, clay soil, poor drainage, proximity to fault zones, urban land use, low vegetation cover) compared to combinations that represent greater stability.
4. THE Assessment_Engine SHALL determine the Risk_Category from the Hazard_Score using the following thresholds: Low for scores in [0, 24], Moderate for scores in [25, 49], High for scores in [50, 74], and Critical for scores in [75, 100].
5. IF the computed Hazard_Score falls outside the range [0, 100], THEN THE Assessment_Engine SHALL treat the result as a computation error, return an error response with a descriptive message, and SHALL NOT persist an incomplete Assessment record.
6. THE Assessment_Engine SHALL ensure that any combination of Input_Parameters that includes at least one high-risk factor (Slope_Angle greater than 45°, Rainfall_Intensity greater than 100 mm/h, Soil_Type of Clay, Drainage_Condition of Poor, or Fault_Zone_Proximity less than 1 km) produces a Hazard_Score greater than 0.
7. IF the Assessment_Engine encounters an internal computation error that is not covered by criteria 5 or 6, THEN THE Assessment_Engine SHALL return an error response with a descriptive message and SHALL NOT persist an incomplete Assessment record.

---

### Requirement 3: Risk Explanation Generation

**User Story:** As a user, I want to see which factors increased or decreased my risk score, so that I can understand the reasoning behind the assessment.

#### Acceptance Criteria

1. WHEN the Assessment_Engine produces a Hazard_Score, THE Assessment_Engine SHALL generate a Risk_Explanation that identifies each Input_Parameter's contribution as risk-increasing, risk-reducing, or neutral.
2. THE Assessment_Engine SHALL include in the Risk_Explanation a plain-language summary statement that contextualises the overall Risk_Category for the assessed site.
3. THE Dashboard SHALL display the Risk_Explanation in a Hazard_Analysis_Card that lists contributing factors with a directional indicator (e.g., upward arrow for risk-increasing, downward arrow for risk-reducing).
4. THE Risk_Explanation SHALL reference each Input_Parameter by its defined name from the Glossary and SHALL NOT use pronouns or ambiguous references.

---

### Requirement 4: Mitigation Recommendation Generation

**User Story:** As a user, I want to receive tailored mitigation recommendations based on my risk level, so that I know what actions to take to reduce landslide susceptibility.

#### Acceptance Criteria

1. WHEN the Assessment_Engine determines a Risk_Category, THE Assessment_Engine SHALL generate a list of one or more Mitigation_Recommendations appropriate to that Risk_Category.
2. THE Assessment_Engine SHALL include "Improve drainage systems" as a Mitigation_Recommendation WHEN the Drainage_Condition input is Poor.
3. THE Assessment_Engine SHALL include "Increase vegetation cover" as a Mitigation_Recommendation WHEN Vegetation_Cover is below 30 percent.
4. THE Assessment_Engine SHALL include "Conduct geotechnical surveys" as a Mitigation_Recommendation WHEN the Risk_Category is High or Critical.
5. THE Assessment_Engine SHALL include "Install retaining structures" as a Mitigation_Recommendation WHEN Slope_Angle exceeds 35 degrees AND Slope_Angle is greater than or equal to 10 degrees, and the Risk_Category is High or Critical.
6. THE Assessment_Engine SHALL include "Restrict construction in vulnerable areas" as a Mitigation_Recommendation WHEN the Risk_Category is Critical.
7. THE Dashboard SHALL display all Mitigation_Recommendations in a dedicated Recommendations_Panel, with each recommendation presented as a distinct, labelled item.

---

### Requirement 5: Assessment Persistence

**User Story:** As a user, I want my assessment results to be saved automatically, so that I can review them later without re-entering data.

#### Acceptance Criteria

1. WHEN the Assessment_Engine successfully completes a computation, THE Assessment_Engine SHALL persist the Assessment record to DynamoDB, including the Input_Parameters, Hazard_Score, Risk_Category, Risk_Explanation, Mitigation_Recommendations, and an ISO 8601 timestamp.
2. THE DynamoDB table SHALL assign each Assessment a unique identifier at the time of creation.
3. THE Assessment_Engine SHALL complete the persistence operation within 3 seconds (inclusive) of completing the computation under normal operating conditions.
4. IF the DynamoDB write operation fails, THEN THE Assessment_Engine SHALL return the computed Assessment result to the Dashboard AND SHALL include a non-blocking warning indicating that the record was not saved.

---

### Requirement 6: Assessment History

**User Story:** As a user, I want to view a list of my previous assessments, so that I can compare results over time and track changes at a site.

#### Acceptance Criteria

1. THE Dashboard SHALL display an Assessment_History table showing all previously saved Assessment records, ordered by timestamp descending (most recent first).
2. THE Assessment_History table SHALL include at minimum the following columns for each record: timestamp, Slope_Angle, Rainfall_Intensity, Soil_Type, Hazard_Score, and Risk_Category.
3. WHEN a user selects a row in the Assessment_History table, THE Dashboard SHALL load the full details of that Assessment into the Hazard_Analysis_Card, Risk_Score_Gauge, and Recommendations_Panel.
4. WHEN the Dashboard initialises, THE Dashboard SHALL attempt to retrieve and display the Assessment_History from the backend API; IF the retrieval fails, THE Dashboard SHALL display any locally available history and show a non-blocking warning indicating that history could not be refreshed.
5. IF the backend API returns no Assessment records, THEN THE Dashboard SHALL display a message indicating that no previous assessments are available.

---

### Requirement 7: Dashboard Visualisation

**User Story:** As a user, I want to see a clear visual summary of my assessment result, so that I can quickly grasp the risk level without reading detailed text.

#### Acceptance Criteria

1. THE Dashboard SHALL display a Risk_Score_Gauge that renders the Hazard_Score as an animated arc or dial within the [0, 100] range.
2. THE Risk_Score_Gauge SHALL render the gauge arc in the colour corresponding to the Risk_Category: green for Low, yellow for Moderate, orange for High, and red for Critical.
3. THE Dashboard SHALL display a Risk_Category_Badge that presents the Risk_Category label (Low, Moderate, High, or Critical) in the corresponding Risk_Category colour.
4. THE Dashboard SHALL apply a dark professional theme using Deep Navy (#0F172A) as the primary background colour, Cyan (#06B6D4) as the primary accent colour, and Emerald (#10B981) as the secondary accent colour.
5. THE Dashboard SHALL render correctly on screen widths from 320 px to 2560 px; on screens narrower than 768 px, THE Dashboard SHALL stack components vertically and MAY hide secondary elements to maintain readability without horizontal scrolling or content overflow.
6. WHEN the Dashboard transitions from displaying a loading state to displaying assessment results, THE Dashboard SHALL animate the Risk_Score_Gauge from 0 to the final Hazard_Score over a duration of 800 ms to 1200 ms.

---

### Requirement 8: PDF Report Generation

**User Story:** As a user, I want to download a PDF report of my assessment, so that I can share the results with colleagues or use them in formal documentation.

#### Acceptance Criteria

1. WHEN a user requests a report for a completed Assessment, THE Dashboard SHALL generate a PDF_Report containing the Input_Parameters, Hazard_Score, Risk_Category, Risk_Explanation, and Mitigation_Recommendations for that Assessment.
2. THE PDF_Report SHALL include the GeoSentinel application name, the Assessment timestamp, and a disclaimer stating that the report represents a preliminary assessment and does not replace a professional geotechnical survey.
3. THE Dashboard SHALL make the PDF_Report available for download within 5 seconds of the user initiating the report request under normal operating conditions.
4. THE PDF_Report SHALL be generated client-side and SHALL NOT require an additional round-trip to the backend API.

---

### Requirement 9: API Communication Security

**User Story:** As a user, I want the application to communicate securely with the backend, so that my assessment data is protected in transit.

#### Acceptance Criteria

1. THE Dashboard SHALL transmit all requests to the backend API exclusively over HTTPS.
2. THE API_Gateway SHALL enforce HTTPS-only access and SHALL reject HTTP requests with a 301 redirect or a 403 response.
3. THE Lambda SHALL validate all incoming request payloads against a defined schema and SHALL return a 400 response with a descriptive error message for any payload that fails validation.
4. THE API_Gateway SHALL apply request throttling with a burst limit of 500 requests per second and a steady-state limit of 200 requests per second to protect backend resources.

---

### Requirement 10: Performance and Availability

**User Story:** As a user, I want the application to respond quickly and remain available, so that I can complete assessments without delays or interruptions.

#### Acceptance Criteria

1. WHEN a user submits valid Input_Parameters, THE Assessment_Engine SHALL return a complete Assessment response within 3 seconds under normal operating conditions.
2. THE CloudFront distribution SHALL serve the Dashboard with a page load time of under 3 seconds on a standard broadband connection (greater than or equal to 10 Mbps).
3. THE Lambda SHALL be configured with sufficient memory and timeout settings to handle concurrent assessment requests without degradation.
4. THE application SHALL be deployed across multiple AWS Availability Zones to support high availability.

---

### Requirement 11: Landing Page

**User Story:** As a first-time visitor, I want to see a professional landing page that explains the application, so that I understand its purpose and how to get started.

#### Acceptance Criteria

1. THE application SHALL provide a public-facing landing page that describes the purpose of GeoSentinel, the supported Input_Parameters, and the types of outputs produced.
2. THE landing page SHALL include a call-to-action element that navigates the user to the Dashboard.
3. THE landing page SHALL apply the same dark professional theme defined in Requirement 7, Criterion 4.
4. THE landing page SHALL be accessible via the root URL of the CloudFront distribution.

---

### Requirement 12: Monitoring and Observability

**User Story:** As a platform operator, I want application errors and performance metrics to be captured automatically, so that I can diagnose and resolve issues without manual instrumentation.

#### Acceptance Criteria

1. THE Lambda SHALL emit structured log entries to Amazon CloudWatch Logs for every Assessment request, including the Assessment identifier, the Risk_Category, and the execution duration.
2. WHEN the Lambda encounters an unhandled exception, THE Lambda SHALL log the full exception details to CloudWatch Logs before returning an error response.
3. THE API_Gateway SHALL be configured to send access logs to CloudWatch Logs, including the HTTP method, resource path, response status code, and request latency for every request.
4. THE CloudWatch configuration SHALL retain Lambda and API_Gateway logs for a minimum of 30 days.
